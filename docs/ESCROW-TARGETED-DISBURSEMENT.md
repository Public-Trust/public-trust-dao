# ЦЕЛЕВОЙ РАСХОД ПОМОЩИ И ESCROW — модель подтверждения

> Нормативно-проектный документ. Производный от [`CONSTITUTION.md`](CONSTITUTION.md)
> (статьи 4, 5, 7), [`PRIORITIES.md`](PRIORITIES.md) и [`ANTI-ABUSE.md`](ANTI-ABUSE.md).
> Это спецификация для смарт-контрактов выплат (Этап 5) и для Audit / Guardian /
> Fairness AI (Этап 6).
>
> **Главный принцип: помощь не выдаётся на руки наличными — она оплачивается
> напрямую поставщику услуги (арендодателю, аптеке, поставщику еды).**
> Тогда «подтверждение целевого расхода» не нужно вымаливать постфактум —
> оно встроено в саму транзакцию и навсегда видно в открытом реестре.

---

## 1. Зачем это нужно

Вопрос «как подтвердить, что деньги ушли на жильё, а не на что-то другое?» —
ключевой для доверия. Классический ответ (собирать чеки после выдачи налички)
плох: его легко подделать, он унижает человека постоянными проверками и создаёт
соблазн нецелевого расхода. Наш подход переворачивает задачу:

- **Не даём деньги — оплачиваем нужду.** Средства движутся от казны к
  поставщику услуги, минуя «руки» получателя помощи.
- **Подтверждение встроено в транзакцию.** В блокчейне навсегда зафиксированы
  сумма, получатель-поставщик, назначение и дата. Проверить может каждый.
- **Это не слежка за человеком.** Публична **транзакция** (кому, сколько, на
  что), а не личность заявителя и не его приватные документы.

Прямо отвечает конституции: статья 3 (открытость и проверяемость), статья 5
(справедливое распределение по нужде, приоритет «угроза потери жилья» = уровень 2
в [`PRIORITIES.md`](PRIORITIES.md)), статья 7 и [`ANTI-ABUSE.md`](ANTI-ABUSE.md)
(защита от нецелевого расхода).

## 2. Три модели целевого расхода

Все три — TESTNET-first. Реальные средства — только после независимого аудита,
multisig 3-из-5 и отдельного одобрения оператора (рельсы безопасности).

### Модель A. Целевой escrow-контракт (основная)

Деньги блокируются в контракте и разблокируются **только** на адрес поставщика
услуги.

```
Заявка одобрена  →  средства заблокированы в escrow  →  release только на
                                                          адрес арендодателя
```

- Получатель помощи **физически не может** вывести средства себе в кэш.
- Поэтапность: для долгой аренды — помесячные транши (release по графику), а не
  вся сумма авансом (механизм «поэтапные выплаты», [`ANTI-ABUSE.md`](ANTI-ABUSE.md) §1).
- Если услуга не оказана/сделка сорвалась — `refund` возвращает средства в казну
  (не получателю), под коллективным контролем.
- Это и есть «смарт-контракт на аренду» простыми словами.

### Модель B. Прямой целевой перевод (earmarked transfer)

Контракт умеет переводить только на **whitelisted-адреса** определённой категории
(например «жильё», «медицина», «питание»).

- Быстрее escrow (нет фазы блокировки), подходит для разовых оплат проверенному
  поставщику.
- Безопасность держится на качестве вайтлиста: кто и как туда вносит поставщиков
  (см. открытые вопросы, §7).

### Модель C. Verified-receipt (мост в оффчейн-мир)

Когда оплата физически проходит вне сети (банк, наличные через проверенного
партнёра), факт целевого расхода фиксируется доказуемо:

- арендодатель/поставщик **подписывает** подтверждение получения (подпись
  кошелька или партнёрский multisig), **или**
- счёт/договор **хешируется** и хеш кладётся в IPFS-манифест и в реестр решений
  (`governance/registry/`), сам документ остаётся приватным.

Публичен и неизменен **факт и его отпечаток**, а не персональные данные.

## 3. Жизненный цикл целевой выплаты (state machine)

```
REQUESTED ──► UNDER_REVIEW ──► APPROVED ──► ESCROWED ──► RELEASED → (поставщику)
                  │                │            │
                  ▼                ▼            ▼
               REJECTED         (appeal)     REFUNDED → (в казну)
                  │                            │
                  ▼                            ▼
               APPEAL ◄───────────────────  DISPUTED
```

- **REQUESTED** — заявка с псевдонимным `case_id`, категорией нужды и приоритетом.
- **UNDER_REVIEW** — коллективная проверка (несколько независимых проверяющих /
  multisig); Fairness/Audit AI помечают аномалии, но **не решают** (статья 9).
- **APPROVED** — решение и его обоснование записываются в публичный реестр.
- **ESCROWED** — средства заблокированы, адрес поставщика зафиксирован.
- **RELEASED** — перевод поставщику (транш или полная сумма), эмитится публичное
  событие для аудита.
- **REFUNDED** — возврат в казну, если услуга не оказана.
- **DISPUTED / APPEAL** — прозрачный путь обжалования ([`ANTI-ABUSE.md`](ANTI-ABUSE.md) §6).

## 4. Привязка к конституции и анти-злоупотреблению

| Требование | Откуда | Как реализуется здесь |
|---|---|---|
| Приоритет нужды | [`PRIORITIES.md`](PRIORITIES.md) | Жильё = уровень 2; ускоряет рассмотрение, но не отключает проверки |
| Поэтапные выплаты | ANTI-ABUSE §1 | Помесячные транши escrow вместо аванса |
| Лимиты | ANTI-ABUSE §2 | Лимит на адрес/период; превышение → усиленная проверка |
| Коллективная проверка | ANTI-ABUSE §3 | Одобрение выше лимита — multisig; нельзя одобрить свою выплату |
| Публичность | CONSTITUTION ст. 3, PRINCIPLES запрет №6 | Все ветви выплат эмитят события; реестр решений публичен |
| Нет единоличного контроля | CONSTITUTION ст. 2, PRINCIPLES запрет №5 | Release/refund/pause — только под коллективным контролем |
| Circuit breaker | ANTI-ABUSE §8 | Пауза выплат при инциденте, без права одного лица вывести средства |
| Апелляции | ANTI-ABUSE §6 | Состояния DISPUTED/APPEAL в модели |
| Приватность заявителя | CONSTITUTION ст. 3 | On-chain только `case_id` и факт; персональные данные — нет |

## 5. Приватность: что публично, а что нет

| Публично (on-chain / реестр) | Приватно (никогда в открытый доступ) |
|---|---|
| Псевдонимный `case_id` | Имя, контакты, адрес заявителя |
| Категория нужды и приоритет | Медицинский диагноз, детали ситуации |
| Сумма и валюта | Сканы паспорта/документов (только их хеш) |
| Адрес поставщика-получателя | — |
| Дата, статус, события | — |
| Хеш счёта/договора (IPFS) | Сам счёт/договор |

Это прямое следствие принципа: проверяема **операция**, а не человек.

## 6. Как объяснять это людям (простым языком)

- **«Мы не даём деньги — мы оплачиваем жильё. И каждый может это проверить.»**
- «Помощь **доходит до цели**, а не растворяется — каждый платёж виден в открытом
  реестре.»
- «Никто не сможет распилить: нет ручной выдачи налички — нет соблазна.»
- «Это **не слежка**: видно, что аренда оплачена, но не видно ваших документов и
  личной истории.»
- «Если сделка сорвалась — деньги возвращаются в общий фонд, а не теряются.»

Запреты соблюдаются буквально ([`PRINCIPLES.md`](PRINCIPLES.md)): нигде не обещаем
доход или выгоду — это оплата нужды, общественное благо, **не инвестиция**.

## 7. Что это значит для Этапа 5 (контракты)

Эскиз интерфейса целевой выплаты (Solidity-псевдокод, для TESTNET):

```solidity
// Псевдокод — спецификация намерения, не финальный контракт.
interface ITargetedDisbursement {
    // Поставщики услуг по категориям (жильё/медицина/питание...).
    function isWhitelistedProvider(address provider, uint8 category) external view returns (bool);

    // Создать целевой escrow: средства уйдут ТОЛЬКО этому поставщику.
    function open(bytes32 caseId, uint8 priorityLevel, address provider, uint256 amount) external returns (uint256 id);

    // Транш поставщику (поэтапные выплаты). Требует коллективного одобрения выше лимита.
    function release(uint256 id, uint256 amount) external; // emits Released(id, provider, amount)

    // Возврат в казну, если услуга не оказана.
    function refund(uint256 id) external;                   // emits Refunded(id, treasury, amount)

    // Аварийная пауза под коллективным контролем (circuit breaker).
    function pause() external;
}
```

Требования к реализации (из [`ANTI-ABUSE.md`](ANTI-ABUSE.md)):

- `release` на сумму выше лимита **обязан** проходить multisig/коллективную проверку.
- Ни один адрес не может одобрить собственную выплату единолично.
- Все ветви (`open`/`release`/`refund`/`pause`) эмитят публичные события.
- Расширить `governance/registry/disbursement.schema.json` полями `provider`,
  `category`, `escrow_id`, `tranche` — чтобы реестр решений и on-chain совпадали.
- Полное покрытие тестами + прогон на testnet (Polygon Amoy предложен в LAUNCH.md)
  «до зелёного». Никаких mainnet/реальных средств без аудита и одобрения оператора.
