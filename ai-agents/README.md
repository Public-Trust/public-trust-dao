# AI-агенты — Public Trust DAO (Этап 6)

> Производный документ. Нормативная основа: [`CONSTITUTION.md`](../docs/CONSTITUTION.md)
> (ст. 9 «Роль искусственного интеллекта»), [`PRINCIPLES.md`](../docs/PRINCIPLES.md),
> [`ANTI-ABUSE.md`](../docs/ANTI-ABUSE.md), [`GOVERNANCE.md`](../docs/GOVERNANCE.md).
>
> **TESTNET-first.** Агенты ничего не двигают и ничем не владеют — они только
> проверяют, считают и докладывают. Любая работа с деньгами/ключами/mainnet — вне
> агентов и только после аудита + одобрения оператора + Safe 3-из-5.

---

## Главный принцип: ИИ служит, но не правит

Конституция, ст. 9: **ИИ не обладает властью, не владеет средствами, служит миссии
и помогает соблюдать конституцию.** AI-агенты — *служебные модули, не органы власти*.

Из этого следуют жёсткие границы для КАЖДОГО агента в этом каталоге:

- **read-only по отношению к средствам.** Ни один агент не подписывает транзакции,
  не двигает казну, не выдаёт/не отзывает права голоса от своего имени. Решает только
  голосование сообщества и Safe-мультисиг (исполнитель воли голосования).
- **сигнал, а не действие.** Находку (нарушение, риск, рассинхрон) агент *показывает*
  людям — он не «исправляет» governance сам. «Красный» от агента — повод для решения
  сообщества, а не команда к автоматическому действию над средствами.
- **проверяемость.** Агенты — открытый код без скрытой логики; их выводы
  воспроизводимы (детерминированно, без обращения к сети и без сторонних зависимостей,
  где это возможно).
- **приватность.** Агенты не публикуют персональные данные заявителей — только
  псевдонимные `case_id` и проверяемые ссылки (дух конституции о достоинстве).

## Восемь агентов (по ст. 9.2)

| Агент | Назначение | Статус |
|-------|-----------|--------|
| **Audit** | Проверяет целостность и прозрачность: реестр (hash-chain), IPFS-манифест, конфиги Safe/Snapshot, тесты контрактов. Единый «зелёный/красный» governance-слоя. | ✅ каркас (`audit_agent.py`) |
| **Guardian** | Следит за рельсами безопасности: нет приватных ключей/секретов в репо, нет mainnet/реальных средств, нет нарушений TESTNET-first. | ✅ каркас (`guardian_agent.py`) |
| **Fairness** | Проверяет справедливость распределения по [`PRIORITIES.md`](../docs/PRIORITIES.md): соблюдён ли приоритет, лимиты/коллективная проверка/поэтапность/приватность. | ✅ каркас (`fairness_agent.py`) |
| **Reputation** | Считает/валидирует веса голоса по [`GOVERNANCE.md`](../docs/GOVERNANCE.md) §2–§3: «1 человек = 1 голос», soulbound, «уникальность ≠ власть». | ✅ каркас (`reputation_agent.py`) |
| **Housing** | Профильный помощник по жилищным кейсам (escrow прямой оплаты поставщику по [`ESCROW-TARGETED-DISBURSEMENT.md`](../docs/ESCROW-TARGETED-DISBURSEMENT.md)). | ✅ каркас (`housing_agent.py`) |
| **Governance** | Помогает с жизненным циклом предложений (формат, кворум, сроки, привязка к конституции), не голосует сам. | ✅ каркас (`governance_agent.py`) |
| **Mediator** | Структурирует споры/апелляции по [`ANTI-ABUSE.md`](../docs/ANTI-ABUSE.md) §6 (обжалуемость санкций, решают люди, независимость, сроки) — структурирует, **не решает**. | ✅ каркас (`mediator_agent.py`) |
| **Documentation** | Следит за двуязычностью RU↔EN (пара + переключатель языка у каждого публичного дока) и целостностью относительных ссылок во всех `.md`. | ✅ каркас (`documentation_agent.py`) |

Реализуем по одному, «до зелёного», без заглушек ради галочки. Каркас открывает
**Audit** — он же сразу полезен: одной командой проверяет весь governance-слой.
Каркас 8/8 закрыт; поверх него — служебный **мета-агент Run-All** (`run_all.py`),
единая точка входа, прогоняющая всех восьмерых и сводящая в один вердикт
(см. раздел «Мета-агент Run-All» ниже).

## Audit-агент — что делает и как запустить

`audit_agent.py` объединяет четыре уже существующих рельс-валидатора в один прогон и
связывает каждую проверку со статьёй конституции, которую она защищает:

| Проверка | Инструмент | Защищает |
|----------|-----------|----------|
| Реестр решений (hash-chain, append-only) | `scripts/registry.py verify` | ст. 3 — открытость, неизменяемость записей |
| IPFS-манифест (CID + sha256) | `scripts/ipfs_manifest.py verify` | ст. 3 — контент и CID не расходятся незаметно |
| Safe multisig 3-из-5 (testnet, без ключей) | `scripts/safe_config.py verify` | ст. 2/4 — нет единоличного владельца; TESTNET-first |
| Snapshot «1 человек = 1 голос» | `scripts/snapshot_config.py verify` | ст. 2 — не плутократия; анти-Сивилла |
| *(опц.)* Тесты смарт-контрактов | `npm test` в `contracts/` | ст. 4/7 — свойства казны/escrow/управления в коде |

Запуск из корня репозитория:

```bash
# Базовый аудит governance-слоя (только стандартная библиотека Python, без сети):
python3 ai-agents/audit_agent.py

# С тестами смарт-контрактов (требует Node/npm в contracts/):
python3 ai-agents/audit_agent.py --with-contracts

# Машиночитаемый отчёт (для CI и других агентов):
python3 ai-agents/audit_agent.py --json
```

Код возврата `0` — всё зелено; `1` — найдено нарушение. «Красный» — это **сигнал**:
агент ничего не исправляет и ничем не распоряжается, решение принимает сообщество.

CI [`.github/workflows/ai-agents.yml`](../.github/workflows/ai-agents.yml) гоняет
базовый аудит на каждый push/PR — публичная гарантия целостности governance-слоя.

**Тест-инвариант** [`test_audit.py`](test_audit.py) доказывает, что Audit **сводит
вердикт честно, а не «зелёный по умолчанию»**: проверка с `exit=1` даёт `fail`,
упавшая/несуществующая команда — `error` (а не молча «pass»), и любой `fail`/`error`
валит общий вердикт в `red`. Сверх синтетики он подсовывает настоящим рельс-
валидаторам реально нарушенный governance-артефакт (Safe-конфиг с mainnet
`chain_id`; Snapshot-конфиг с плутократической стратегией) во временной копии и
доказывает, что Audit поднимает это в «красный», а на здоровом репозитории — «зелёный».

## Guardian-агент — что делает и как запустить

`guardian_agent.py` — отдельный явный **сканер рельс безопасности по всему дереву
репозитория** (а не по одному конфигу, как делают `safe_config.py`/`snapshot_config.py`).
Проходит по git-отслеживаемым файлам (то, что реально опубликовано) и проверяет:

| Проверка | Что ловит | Защищает |
|----------|-----------|----------|
| `secrets-tracked` | закоммичен ли секрет/ключ (`.env`, `*.key`, `*.pem`, `keystore/`, `wallet*.json`, `*_private*`, файлы состояния пульса) | рельс «секреты никогда в репо» |
| `gitignore-guards` | прикрывает ли `.gitignore` `.env` и `logs/` (состояние пульса) | рельс «секреты не в репо»; пульс неприкосновенен |
| `no-mainnet` | нет ли mainnet `chain_id` ни в одном JSON-конфиге | рельс TESTNET-first |
| `no-key-material` | нет ли приватных ключей в тексте (64-hex вне хэш-полей; присваивание `private key`/`mnemonic`/`seed` реальным значением) | ст. 2/4 — нет владельца с ключом от реальных денег |

Чтобы не было ложных срабатываний на легитимных `sha256`/CID реестра и манифеста,
64-hex считается ключом **только вне** полей-хэшей (`hash`/`prev_hash`/`sha256`/…), а
ссылки на окружение (`process.env.PRIVATE_KEY`) не считаются утечкой. Контекст хэша
распознаётся и в человекочитаемых текстах, в том числе по русским словам-подсказкам
(«отпечаток», «хэш»/«хеш») — например, строка «Текущий отпечаток журнала: `<64-hex>`»
в `CHANGELOG.md` не принимается за приватный ключ (`PTD-0115`).

```bash
python3 ai-agents/guardian_agent.py          # человекочитаемый отчёт
python3 ai-agents/guardian_agent.py --json    # машиночитаемый (для CI/других агентов)
```

Код возврата `0` — рельсы целы; `1` — найдено нарушение (это **сигнал**, не действие).

**Тест-инвариант** [`test_guardian.py`](test_guardian.py) доказывает, что Guardian
работает, а не «зелёный по умолчанию»: на каждое нарушение (закоммичен `.env`/ключ,
mainnet `chain_id`, 64-hex-ключ, присваивание секрета, дырявый `.gitignore`) агент
обязан вернуть «красный», а на чистом дереве с настоящими sha256-хэшами — «зелёный».

CI [`.github/workflows/ai-agents.yml`](../.github/workflows/ai-agents.yml) на каждый
push/PR прогоняет Audit + тест-инвариант Guardian + сам Guardian.

## Fairness-агент — что делает и как запустить

`fairness_agent.py` проходит по записям публичного реестра типа `disbursement`
(`governance/registry/`) и проверяет КАЖДУЮ выплату на соответствие нормам
справедливого распределения и антизлоупотребления. Шкала приоритетов читается
**прямо из** [`docs/PRIORITIES.md`](../docs/PRIORITIES.md) (а не зашита в код) —
так заодно проверяется, что код и норматив не разошлись.

| Проверка | Что требует | Защищает |
|----------|-------------|----------|
| `priority-valid` | уровень приоритета входит в шкалу 1..10 из PRIORITIES.md | ст. 5 — приоритет по характеру нужды, а не по личности |
| `safeguards` | приоритет НЕ отключает антизлоупотребление: `limit_ok`/`collective_review`/окно апелляции | ст. 7 — высокий приоритет ускоряет, но не отменяет защиту (PRIORITIES, правило 2) |
| `collective-review` | выплату подтверждает ≥2 независимых проверяющих | ст. 7 — нельзя одобрить выплату единолично (ANTI-ABUSE §3) |
| `staged-payments` | поэтапность валидна (`1 <= index <= of`) | ст. 7 — дробление снижает риск (ANTI-ABUSE §1) |
| `applicant-privacy` | в записи нет персональных данных (имя/e-mail/телефон/…), только псевдонимный `case_id` | ст. 5/6 — достоинство и приватность заявителя |

```bash
python3 ai-agents/fairness_agent.py          # человекочитаемый отчёт
python3 ai-agents/fairness_agent.py --json    # машиночитаемый (для CI/других агентов)
```

Код возврата `0` — распределение справедливо; `1` — найдено нарушение (это
**сигнал** сообществу, не действие: агент ничего не исправляет и не распоряжается).

**Тест-инвариант** [`test_fairness.py`](test_fairness.py) доказывает, что Fairness
работает, а не «зелёный по умолчанию»: на каждое нарушение (приоритет вне шкалы,
сняты лимиты/окно апелляции, единоличное одобрение, сломанная поэтапность,
персональные данные/e-mail в записи) агент обязан вернуть «красный», а на
корректной выплате и на пустом реестре — «зелёный» без ложных срабатываний.

CI [`.github/workflows/ai-agents.yml`](../.github/workflows/ai-agents.yml) на каждый
push/PR прогоняет Audit + Guardian (+тест) + Fairness (+тест).

## Reputation-агент — что делает и как запустить

`reputation_agent.py` — read-only статический разбор: доказывает, что модель
«1 человек = 1 голос» сохранена **в коде** ([`contracts/contracts/Reputation.sol`](../contracts/contracts/Reputation.sol))
и **в настройках голосования** ([`governance/snapshot/space.json`](../governance/snapshot/space.json)),
а не только «на словах» по [`docs/GOVERNANCE.md`](../docs/GOVERNANCE.md) §2–§3.

| Проверка | Что требует | Защищает |
|----------|-------------|----------|
| `soulbound` | у бейджа нет функций перевода (`transfer`/`approve`/…) — голос непередаваем | ст. 2 / §2 — право голоса не продаётся и не скупается |
| `bounded-weight` | `votingUnits`: не участник → 0; участник → `1 + min(points, cap)`, коридор `[1..1+cap]` | ст. 2 / §2 — власть денег невозможна (запрет №5) |
| `no-funds` | слой репутации не двигает средства (`payable`/`.transfer`/`.call{value}`/…) | ст. 9 / §3 — «уникальность ≠ власть» |
| `roles-separated` | `verifier` только mint/revoke бейджа; `governor` только параметры; роли не смешаны | §3 — кто подтверждает человека, тот не правит управление |
| `off-chain-equal` | стратегия Snapshot = равный `ticket` value=1 (не плутократия по балансу), допуск только участникам | ст. 2 / §2,§4 — офчейн-сигнал тоже «1 человек = 1 голос» |

```bash
python3 ai-agents/reputation_agent.py          # человекочитаемый отчёт
python3 ai-agents/reputation_agent.py --json    # машиночитаемый (для CI/других агентов)
```

Код возврата `0` — модель голоса цела; `1` — найдена угроза (это **сигнал**
сообществу, не действие: агент ничего не исправляет и не распоряжается).

**Тест-инвариант** [`test_reputation.py`](test_reputation.py) доказывает, что
Reputation работает, а не «зелёный по умолчанию»: на каждую угрозу (добавлена
функция перевода, вес без потолка/по балансу, контракт двигает средства, смешение
ролей, плутократическая стратегия Snapshot, `ticket` value≠1, допуск не только
участникам) агент обязан вернуть «красный», а на корректных контракте+настройках —
«зелёный» без ложных срабатываний (включая отсев упоминаний в комментариях).

CI [`.github/workflows/ai-agents.yml`](../.github/workflows/ai-agents.yml) на каждый
push/PR прогоняет Audit + Guardian (+тест) + Fairness (+тест) + Reputation (+тест)
+ Housing (+тест) + Documentation (+тест) + Governance (+тест) + Mediator (+тест).

## Housing-агент — что делает и как запустить

`housing_agent.py` — профильный помощник по жилищным кейсам. Read-only доказывает,
что главный принцип целевого расхода — **«помощь не выдаётся на руки, она
оплачивается напрямую поставщику услуги (арендодателю)»** — заложен **в коде**
([`contracts/contracts/Disbursement.sol`](../contracts/contracts/Disbursement.sol)),
а не только описан в [`docs/ESCROW-TARGETED-DISBURSEMENT.md`](../docs/ESCROW-TARGETED-DISBURSEMENT.md);
и проверяет жилищные записи реестра (`category=housing`).

Часть A — инварианты контракта целевого escrow:

| Проверка | Что требует | Защищает |
|----------|-------------|----------|
| `release-to-provider-only` | `release(id, amount)` не принимает адрес получателя; транш уходит строго на `c.provider` кейса | ст. 5 / ESCROW §2 — оплата напрямую поставщику, не «на руки» |
| `provider-fixed` | адрес поставщика фиксируется в `open`, не переназначается (нет `setProvider`/`.provider =`) | ст. 5 / ESCROW §3 — единственный возможный получатель задан заранее |
| `refund-to-treasury` | `refund` возвращает остаток в `treasury`, а не получателю/поставщику | ст. 7 / ESCROW §2 — услуга не оказана → средства не теряются, идут в фонд |
| `tranche-limit` | `release` ограничивает один транш потолком `maxRelease` | ANTI-ABUSE §1–§2 — поэтапные выплаты/лимиты снижают риск |
| `guardian-cannot-move` | средства двигает только `executor`; `guardian` лишь ставит паузу (без движения средств) | ст. 2/9 — «безопасность ≠ власть» |

Часть B — жилищные записи реестра (`category=housing`):

| Проверка | Что требует | Защищает |
|----------|-------------|----------|
| `targeted-escrow` | у записи есть `provider` + `escrow_id` (привязка к on-chain escrow) | ст. 3/5 / ESCROW §2 — оплата напрямую поставщику, видна в реестре |
| `provider-onchain` | `provider` — валидный ненулевой адрес `0x`+40 hex | ст. 5 / ESCROW §5 — публичен адрес поставщика, не личность |
| `category-priority` | у `category=housing` уровень = «housing_loss» из [`PRIORITIES.md`](../docs/PRIORITIES.md) (читается из документа) | ст. 5 — приоритет по характеру нужды (жильё), а не по личности |

```bash
python3 ai-agents/housing_agent.py          # человекочитаемый отчёт
python3 ai-agents/housing_agent.py --json    # машиночитаемый (для CI/других агентов)
```

Код возврата `0` — модель целевого расхода цела; `1` — найдено отклонение (это
**сигнал** сообществу, не действие: агент ничего не исправляет и не распоряжается).

**Тест-инвариант** [`test_housing.py`](test_housing.py) доказывает, что Housing
работает, а не «зелёный по умолчанию»: на каждое отклонение (release с адресом-
параметром, появился `setProvider`, `refund` получателю вместо казны, снят потолок
транша, `release` под `guardian`, запись без `escrow_id`, битый/нулевой адрес
поставщика, неверный приоритет жилья) агент обязан вернуть «красный», а на
корректных контракте+записях — «зелёный» без ложных срабатываний (включая отсев
упоминаний в комментариях и игнор не-жилищных записей).

CI прогоняет Housing (+тест) в том же workflow.

## Documentation-агент — что делает и как запустить

`documentation_agent.py` превращает в машинную проверку два правила проекта:
указание оператора **«вся документация двуязычна (RU↔EN)»** и конституционную
**проверяемость** (ст. 3) / **открытость** (ст. 6). Read-only по
git-отслеживаемым `.md` (по тому, что реально опубликовано):

| Проверка | Что требует | Защищает |
|----------|-------------|----------|
| `bilingual-pairs` | у каждого публичного дока есть пара RU↔EN (правило пар по пути) | ст. 6 — открытость/понятность; «вся документация двуязычна» |
| `language-switcher` | вверху дока корректный переключатель `[Русский]·[English]`, ведущий к парному файлу | ст. 6 — доступность на обоих языках |
| `link-integrity` | все относительные ссылки в `.md` ведут к существующему файлу/каталогу | ст. 3 — проверяемость; документ без битых ссылок реально читаем |
| `anchor-integrity` *(мягкая)* | якорь ссылки `ФАЙЛ.md#раздел` (и `#раздел` внутри файла) ведёт к реальному заголовку (слаг по правилу GitHub) | ст. 3 — проверяемость; ссылки не гниют при переименовании разделов |

| `constitutional-prohibitions` *(мягкая)* | публичные тексты (`.md` + витрина `web/*.html`) не дают запрещённых обещаний: доходность/прибыль, инвестиция/пирамида, плата за привлечение (рефералы) — наши дисклеймеры «это НЕ инвестиция» и перечни запретов не считаются нарушением | ст. 3/6 + `PRINCIPLES.md` «Конституционные запреты» для публичных текстов |
| `see-also-targets` *(мягкая)* | каждый адрес карт «См. также» платформы (`RELATED`/`RELATED_ACTIONS` в `SeeAlso.tsx`) существует в текстах `i18n` (`t.learn`/`t.screens`) — перекрёстная ссылка не «отвалится» молча при переименовании экрана | ст. 3/6 — проверяемость/понятность (`PTD-0105`) |

| `mirror-doc-link` *(мягкая)* | каждый экран-зеркало платформы (пересказ нормативного дока простыми словами) ссылается на свой первоисточник в `docs/` — пересказ можно сверить с нормой, зеркало не «уезжает» от документа | ст. 3/6 — проверяемость/понятность (`PTD-0040`) |

| `mirror-doc-showcase` *(мягкая)* | парная к `mirror-doc-coverage`: заведённый экран-зеркало из `MIRROR_DOCS` (страница есть) показан в витрине `t.learn` — иначе построенный экран-объяснение «спрятан» от человека мимо главной | ст. 3/6 — проверяемость/понятность (`PTD-0110`) |
| `mirror-doc-distinct` *(мягкая)* | у каждого экрана-зеркала свой отдельный документ: два разных ключа `MIRROR_DOCS` не указывают на ОДИН и тот же `docs/X.md` — иначе один документ получает два зеркала, а другой остаётся вовсе без своего (почти всегда опечатка в карте) | ст. 3/6 — проверяемость/понятность (`PTD-0111`) |
| `mirror-doc-original` *(мягкая)* | путь-значение карты `MIRROR_DOCS` ведёт на RU-первоисточник в `docs/`, а не на перевод (`docs/en/…` или `*.en.md`) — иначе пересказ сверяется с переводом, а не с нормой, и зеркало «уедет» при расхождении RU↔EN незаметно | ст. 3/6 — проверяемость/понятность (`PTD-0111`) |
| `mirror-doc-normalized` *(мягкая)* | путь-значение карты `MIRROR_DOCS` записан в нормализованной форме ровно `docs/<ИМЯ>.md` (без хвостовых пробелов, без `./`/`../`, оканчивается на `.md`, не указывает на каталог/подкаталог) — «грязная» запись может молча обойти `mirror-doc-exists`/`mirror-doc-original` | ст. 3/6 — проверяемость/понятность (`PTD-0112`) |
| `mirror-doc-bilingual` *(мягкая)* | у нормативного документа из карты (`docs/X.md`, который реально есть) существует EN-зеркало `docs/en/X.md` — иначе экран ведёт на RU-первоисточник, а англоязычный человек, перейдя в норму, упрётся в тупик; связывает карту зеркал с правилом двуязычности | ст. 3/6 — проверяемость/понятность (`PTD-0112`) |
| `mirror-doc-slug` *(мягкая)* | ключ карты `MIRROR_DOCS` записан как валидный адрес-слаг `<строчные-с-дефисами>` (только `a-z`, цифры и одиночные дефисы, без заглавных, пробелов, слэшей, ведущего/хвостового и двойных дефисов) — иначе ключ перестаёт быть рабочим адресом экрана `/<ключ>/` и карта тихо рассинхронится с витриной `t.learn` | ст. 3/6 — проверяемость/понятность (`PTD-0114`) |
| `mirror-doc-learn-slug` *(мягкая)* | парная к `mirror-doc-slug` с другой стороны: каждый адрес самой витрины `t.learn` (`lib/i18n.ts`) записан как валидный слаг ровно `/<строчные-с-дефисами>/` (ведущий и хвостовой слэш, между ними только `a-z`, цифры и одиночные дефисы) — иначе кривой адрес витрины (`/Manifesto/`, `/direct help/`) тихо рассинхронит её с картой `MIRROR_DOCS`, а `mirror-doc-slug` смотрит только ключи и этого не ловит | ст. 3/6 — проверяемость/понятность (`PTD-0117`) |
| `mirror-doc-set-match` *(мягкая)* | прямая сверка двух множеств слагов: адреса витрины `t.learn` и ключи карты `MIRROR_DOCS` описывают одно и то же множество экранов. Показывает разом, что есть только в витрине и что только в карте — рассинхрон объясняется одной строкой, а не вычитанием отчётов `mirror-doc-coverage`/`mirror-doc-showcase`. Ключ без построенной страницы со стороны карты в сверку не входит («ещё не сделано», а не «разошлось») | ст. 3/6 — проверяемость/понятность (`PTD-0118`) |
| `mirror-doc-learn-built` *(мягкая)* | у каждого адреса `/slug/` витрины `t.learn` (`lib/i18n.ts`) реально построена страница экрана `platform/app/<slug>/page.tsx`. Закрывает пробел: `mirror-doc-coverage`/`mirror-doc-set-match` сверяют адрес с картой, а `link-integrity` смотрит только `.md` — никто не ловил «адрес в витрине есть, а самого экрана ещё нет», и человек упёрся бы с главной в 404. Нет `i18n.ts` или каталога платформы → молчит | ст. 3/6 — проверяемость/понятность (`PTD-0119`) |
| `see-also-symmetric` *(мягкая)* | связь «См. также» между экранами-объяснениями взаимна: если экран A ведёт на B в `RELATED`, то и B ведёт обратно на A — у человека есть обратный путь, нет тупиков. Несимметрия допустима (хабы вроде словаря/«Как решаем»), поэтому только предупреждение | ст. 3/6 — проверяемость/понятность (`PTD-0106`) |

> Проверки `glossary-coverage` и `glossary-no-dead` — **мягкие**: они только
> предупреждают (чтобы глоссарий не отставал от документов и не оброс лишним) и
> **не роняют вердикт** — «зелёно» остаётся «зелёным». Число предупреждений видно
> в отчёте (поле `warnings`). `glossary-no-dead` сознательно консервативна (термин
> ищется по многим ключам: основная фраза, варианты из скобок, отдельные значимые
> слова) — лучше промолчать, чем дать ложную тревогу.

**Правило пар** (выводится из пути, а не зашито пофайлово):
`docs/NAME.md` ↔ `docs/en/NAME.md`; `<dir>/README.md` ↔ `<dir>/README.en.md`;
корневой `NAME.md` ↔ `NAME.en.md` (например `REPO-STRUCTURE`, `AUTHORS`).
Служебные одноязычные файлы (`BUILDER.md`, `LAUNCH.md`, `PROGRESS.md`,
`DECISIONS.md`, `comms/*`) исключены by design — это рабочая кухня контура, а не
публичная документация фонда. Внешние ссылки (`http`/`mailto`/якоря) и содержимое
блоков кода ```…``` в проверке ссылок не участвуют — без ложных срабатываний.

```bash
python3 ai-agents/documentation_agent.py          # человекочитаемый отчёт
python3 ai-agents/documentation_agent.py --json    # машиночитаемый (для CI/других агентов)
```

Код возврата `0` — документация цела и двуязычна; `1` — найдено расхождение (это
**сигнал** сообществу, не действие: агент не дописывает переводы и не правит ссылки).

**Тест-инвариант** [`test_documentation.py`](test_documentation.py) доказывает, что
Documentation работает, а не «зелёный по умолчанию»: на каждое расхождение (нет
EN-зеркала, нет RU-оригинала, нет/битый/перепутанный переключатель, битая
относительная ссылка) агент обязан вернуть «красный», а на чистой двуязычной
документации — «зелёный» без ложных срабатываний (служебные одноязычные файлы,
внешние ссылки и код-блоки не должны его ронять). При первом прогоне агент сразу
поймал реальный пробел — у `governance/ipfs/README.md` и
`governance/registry/README.md` не было EN-зеркал; они добавлены (теперь 8/8).

CI прогоняет Documentation (+тест) в том же workflow.

## Governance-агент — что делает и как запустить

`governance_agent.py` превращает в машинную проверку **жизненный цикл предложения**
из [`GOVERNANCE.md`](../docs/GOVERNANCE.md): чтобы конфигурация управления не
разъезжалась с конституцией молча. Read-only по двум JSON-конфигам управления
([`governance/snapshot/space.json`](../governance/snapshot/space.json) и
[`governance/safe/safe.config.json`](../governance/safe/safe.config.json)), без сети.
**Сам НЕ голосует, ничего не вносит и ничем не распоряжается.**

| Проверка | Что требует | Защищает |
|----------|-------------|----------|
| `one-person-one-vote` | стратегия голоса = `ticket` value=1; ни одной плутократической (вес по балансу) | ст. 2 + запрет №5 — голос даёт уникальность человека, а не деньги |
| `timed-vote` | у голосования задан положительный срок (`delay` и `period` > 0) | GOVERNANCE §7 — «кворум+срок», окно ограничено во времени |
| `off-chain-signal` | `off_chain_signaling=true` и все типы предложений `binding=false` | ст. 4 + §5 — Snapshot обсуждает, исполняет Safe/Timelock после голосования |
| `proposal-binding` | типы про деньги/конституцию привязаны к нужным докам и флагам | ст. 5 + §7–§8 — распределение и поправки строго в рамках норм |
| `multisig-not-sole` | порог Safe ≥2 и строго меньше числа владельцев | ст. 5 + §5 — никто не распоряжается средствами единолично (3-из-5) |
| `lifecycle-links` | все `docs/`–/`governance/`-ссылки конфигов ведут к существующему файлу | ст. 3 — проверяемость: цикл привязан к реальным докам |

**Привязка типов** (`proposal-binding`): тип `disbursement-direction` обязан ссылаться
на [`PRIORITIES.md`](../docs/PRIORITIES.md) **и** [`ANTI-ABUSE.md`](../docs/ANTI-ABUSE.md);
тип `constitution-amendment` — на [`CONSTITUTION.md`](../docs/CONSTITUTION.md) **и** нести
`requires_supermajority=true` (поправка к конституции — супербольшинство, ст. 10).

```bash
python3 ai-agents/governance_agent.py          # человекочитаемый отчёт
python3 ai-agents/governance_agent.py --json    # машиночитаемый (для CI/других агентов)
```

Код возврата `0` — конфигурация управления согласована с конституцией; `1` —
найдено расхождение (это **сигнал** сообществу, не действие: агент не правит
конфиг и не голосует).

**Тест-инвариант** [`test_governance.py`](test_governance.py) доказывает, что
Governance работает, а не «зелёный по умолчанию»: на каждое отклонение (вес голоса
по балансу, `ticket` value≠1, плутократия в `validation`, нулевой срок, тип
`binding=true`, распределение/поправка без привязки к нормам, порог мультисига
`=1` или `=числу владельцев`, битая ссылка, отсутствие самого `space.json`) агент
обязан вернуть «красный», а на корректных конфигах — «зелёный» без ложных
срабатываний (26/26). CI прогоняет Governance (+тест) в том же workflow.

## Mediator-агент — что делает и как запустить

`mediator_agent.py` превращает в машинную проверку **право на апелляцию** (механизм
№6 «Апелляции» из [`ANTI-ABUSE.md`](../docs/ANTI-ABUSE.md)): чтобы прозрачный путь
обжалования отказа/санкции не остался лозунгом и не разъехался с конституцией
молча. Read-only по артефакту процесса
[`governance/mediation/dispute-process.json`](../governance/mediation/dispute-process.json),
без сети. **Mediator структурирует спор, но НИКОГДА его не решает** (ст. 9.2):
решают независимые люди, агент лишь проверяет форму процесса и докладывает.

| Проверка | Что требует | Защищает |
|----------|-------------|----------|
| `appeal-for-every-sanction` | у каждой канонической санкции (отказ, репутация, заморозка, исключение) есть обжалование к существующей стадии | §6 — прозрачный путь обжалования отказа/санкции |
| `mediator-not-decider` | решающую стадию ведут люди (`min_deciders` ≥2), не ИИ/медиатор/автомат и не один человек | ст. 9.2 + §3 — ИИ не орган власти; решает коллектив |
| `independent-review` | первая решающая стадия от точки входа имеет `decider` ≠ `issued_by` санкции | §3 — независимость; нельзя пересматривать своё решение |
| `valid-lifecycle` | ровно одна начальная, есть терминальная, переходы целы, нет тупиков/сирот, всё достижимо | ст. 3 — проверяемость: целостный автомат |
| `bounded-timelines` | у каждой не-терминальной стадии `deadline_seconds` > 0 | ст. 3 + §6 — апелляция не зависает навсегда |
| `process-links` | все `docs/`–/`governance/`-ссылки процесса ведут к существующему файлу | ст. 3 — привязка к реальным нормам |

**Жизненный цикл** (артефакт): `intake` (приём, структурирует Mediator) → `review`
(решают ≥2 независимых) → `escalation` (совет хранителей ≥3) → `resolved`
(терминал, фиксация в реестре). Решение — только у людей и только коллективно;
Mediator-агент допустим лишь на не-решающих стадиях.

```bash
python3 ai-agents/mediator_agent.py          # человекочитаемый отчёт
python3 ai-agents/mediator_agent.py --json    # машиночитаемый (для CI/других агентов)
```

Код возврата `0` — процесс обжалования согласован с конституцией; `1` — найдено
расхождение (это **сигнал** сообществу, не действие: агент не правит процесс и не
решает споры).

**Тест-инвариант** [`test_mediator.py`](test_mediator.py) доказывает, что Mediator
работает, а не «зелёный по умолчанию»: на каждое отклонение (необжалуемая/
отсутствующая санкция, `appeal_entry` в никуда, спор решает ИИ или один человек,
самопересмотр `decider==issued_by`, тупик/несуществующий переход/нет терминала/две
начальные стадии, нулевой срок, битая ссылка, отсутствие самого
`dispute-process.json`) агент обязан вернуть «красный», а на корректном процессе —
«зелёный» без ложных срабатываний (26/26). CI прогоняет Mediator (+тест) в том же
workflow. **Этим закрыт каркас всех восьми AI-агентов (8/8).**

## Мета-агент Run-All — единая точка входа

Когда каркас всех восьми агентов закрыт, нужна одна команда, которая прогоняет их
все и сводит в один вердикт. Это `run_all.py` — **служебный мета-модуль** (ст. 9):
сам ничего не проверяет по существу и тем более ничем не распоряжается, только
запускает восьмерых и собирает их `--json`-отчёты в общий «зелёно/красно».

Зачем:

- **локальный self-check одной командой** вместо восьми ручных запусков;
- **CI сворачивается с ~15 шагов до двух** (`--with-tests` гоняет и сами агенты, и
  их тест-инварианты);
- **машиночитаемая сводка** (`--json`) для дашбордов и других агентов.

```bash
# Прогнать все восемь агентов, свести в один вердикт (только стандартная библиотека):
python3 ai-agents/run_all.py

# Дополнительно прогнать тест-инварианты агентов (test_*.py — «красное реально ловится»):
python3 ai-agents/run_all.py --with-tests

# Пробросить в Audit тесты смарт-контрактов (требует Node/npm):
python3 ai-agents/run_all.py --with-contracts

# Машиночитаемый отчёт:
python3 ai-agents/run_all.py --json

# Записать компактный машиночитаемый «статус-светофор» в файл-артефакт:
python3 ai-agents/run_all.py --with-tests --status-out governance/status/run_all_status.json
```

Код возврата `0` — всё зелено; `1` — хотя бы один агент (или тест-инвариант при
`--with-tests`) красный. Мета-агент считает агента «красным» не только по его
вердикту, но и при любой аномалии — невалидный JSON (агент упал) или
рассогласование «verdict=green, а код возврата ≠ 0». «Красный» — **сигнал**, а не
действие: мета-агент ничего не исправляет.

Помимо восьми агентов мета-агент прогоняет ещё и **стража структуры**
([`structure_guard.py`](structure_guard.py), см. ниже) и выводит его отдельной
графой в сводке — `страж 10/10, мягких предупреждений стража: N`. Его
hard-«красное» (нарушение обязательного стандарта качества) валит общий вердикт
наравне с агентами, а его **мягкие предупреждения** (severity=soft — например, у
шага CI нет человеческого имени) выводятся числом И **построчно** (под графой
стража — список `⚠ [проверка] шаг: проблема`) и **не валят** сборку: так
операторская сводка не теряет подсказок к читаемости, объясняет их без отдельного
запуска стража, но и не краснеет из-за них. В `--json` те же строки лежат в ключе
`guard_warning_lines`. Если стража в каталоге нет — графа не показывается, на
вердикт не влияет.

Флаг **`--status-out PATH`** дополнительно пишет компактный машиночитаемый
**«статус-светофор»** в файл-артефакт (помощник `build_status`): общий вердикт,
счёт агентов/тестов, сводка стража (число и строки мягких предупреждений) и по
строке на каждого агента. Артефакт **детерминирован** — без времени по стенным
часам, поэтому в git «грязнеет» только при смене вердикта/счёта; поле
`schema_version` версионирует формат. Канонический артефакт лежит в
[`governance/status/run_all_status.json`](../governance/status/) (с двуязычным
README) — это основа публичного статус-индикатора без внешних сервисов. Пишется
при любом вердикте, в т.ч. `red` (индикатор честно показывает и красный свет).

**Тест-инвариант** [`test_run_all.py`](test_run_all.py) доказывает, что сведение
работает, а не «зелёное по умолчанию»: на фейковых агентах (зелёный/красный/упавший/
аномальный), фейковых тестах и фейковом страже (мягкие предупреждения не валят,
hard-«красное» валит, отсутствие стража неприменимо; сами строки предупреждений
показываются под графой стража; статус-артефакт пишется детерминированно и при
красном вердикте тоже) проверяет, что красное реально сводится в красное,
а зелёное не падает ложно (39/39). CI прогоняет
`test_run_all.py` + `run_all.py --with-tests` — два шага вместо пятнадцати.

## Страж структуры (Structure-Guard) — стережёт стандарты качества каталога

К Этапу 6 мы установили стандарты качества, но до сих пор их никто не стерёг
автоматически — их можно было нечаянно нарушить и не заметить. `structure_guard.py`
делает их машинной проверкой (ст. 9 — служебный модуль, ничего не правит):

| Проверка | Что стережёт |
|----------|--------------|
| `agents-have-invariants` | У каждого `*_agent.py` есть парный `test_<имя>.py` — агент доказывает, что ловит «красное», а не «зелёный по умолчанию» (так был закрыт пробел Audit в сессии 33). |
| `no-orphan-tests` | У каждого `test_*.py` есть исходный модуль (`<имя>_agent.py` или `<имя>.py`) — тест-файл не висит после переименования/удаления. |
| `sol-parsing-centralized` | Ни один `*_agent.py` не заводит локальную копию помощников разбора `.sol` (`strip_solidity_comments`/`function_body`/…) — их импортируют из общего [`solidity_scan.py`](solidity_scan.py), а не размножают (иначе копии тихо разойдутся). |
| `run-all-covers-all` | Каждый `*_agent.py` есть в списке `AGENTS` мета-агента [`run_all.py`](run_all.py), а каждый `test_*.py` — в списке `TESTS` (и наоборот: нет висячих ссылок) — новый агент или тест нельзя завести в обход общего прогона CI. |
| `trigger-paths-include-agents` | Триггер-пути воркфлоу (`on.push.paths` и `on.pull_request.paths`) содержат `ai-agents/**` — иначе правка агента не запустит CI, и покрытие обходится ещё раньше, на уровне триггера, а не команды. |
| `ci-has-required-steps` | Сам воркфлоу [`.github/workflows/ai-agents.yml`](../.github/workflows/ai-agents.yml) содержит каждую команду из единого списка обязательных шагов CI (`REQUIRED_WORKFLOW_COMMANDS`): сейчас это `run_all.py --with-tests` (прогон всех агентов и их тест-инвариантов — без `--with-tests` или в обход они не прогонятся) и `test_run_all.py` (доказывает, что сведение в общий вердикт исправно: красное→красное). Обобщает прежние точечные проверки `ci-calls-run-all` и `ci-runs-test-run-all`: новый обязательный шаг CI — одна строка в списке, без размножения почти одинаковых проверок. |
| `ci-required-cmd-own-step` | У каждой обязательной команды есть СВОЙ отдельный шаг `run:`. Проверка выше видит команду «где угодно» в файле — включая комментарий или название шага; эта смотрит сами тела шагов `run:` и требует, чтобы каждая команда (а) реально запускалась хотя бы одним шагом и (б) не делила шаг с другой. Иначе провал одной маскирует другую: в одном `run:` с `&&` вторая команда после падения первой не исполнится, а оставшаяся лишь в комментарии — не исполнится вовсе. Свой шаг — свой код возврата, свой провал на виду. |
| `ci-step-has-name` *(мягкая — предупреждает, не валит CI)* | У шага `run:` обязательной команды есть человеческое `- name:`. Развитие предыдущей проверки в сторону читаемости: без имени упавший шаг показан в логах GitHub Actions голой командой (`python3 ai-agents/run_all.py --with-tests`), а не по-человечески («Прогнать всех агентов»). Это не ломает покрытие, поэтому проверка лишь **предупреждает** (severity=soft) и не краснит вердикт — подсказка улучшить читаемость ради людей (ст. 3 «понятность»). |
| `ci-step-name-unique` *(мягкая — предупреждает, не валит CI)* | Два шага `run:` не носят ОДНО И ТО ЖЕ `- name:`. Развитие предыдущей: имя у шага есть, но если оно повторяется, упавший шаг в логах GitHub Actions читается двусмысленно («какой из двух одноимённых?»). Тоже лишь **предупреждает** (severity=soft) и не краснит вердикт. Безымянные шаги пропускает — о них заботится `ci-step-has-name`. |
| `ci-step-has-body` *(мягкая — предупреждает, не валит CI)* | У шага с `- name:` есть тело `run:`/`uses:` (он не пустой). Развитие предыдущей в сторону честности карты шагов: если элемент списка несёт `- name:`, но НИ `run:`, НИ `uses:`, это почти всегда опечатка отступа — тело шага «выпало» из него, и шаг ничего не делает, хотя в интерфейсе GitHub Actions выглядит настоящим. Тоже лишь **предупреждает** (severity=soft): реальный пропуск обязательной команды поймают hard-проверки выше, а эта бережёт читаемость карты CI ради людей (ст. 3 «понятность»). |
| `issue-form-labels-defined` | Каждая метка, на которую ссылается форма обращения (`labels:` в [`.github/ISSUE_TEMPLATE/*.yml`](../.github/ISSUE_TEMPLATE) — `bug`/`idea`/`governance`/`safety`), описана в каталоге меток [`.github/labels.yml`](../.github/labels.yml). Раньше цвет и описание этих меток жили только в настройках GitHub и заводились вручную — невоспроизводимо. Теперь источник правды — открытый файл, а страж краснеет, если форма ссылается на метку, которой в каталоге нет (заведена руками/опечатка). Нарушение называет ровно пропавшую метку и форму. |

```bash
python3 ai-agents/structure_guard.py          # человекочитаемый отчёт
python3 ai-agents/structure_guard.py --json    # машиночитаемый отчёт
```

**Тест-инвариант** [`test_structure_guard.py`](test_structure_guard.py) на
отравленных временных каталогах (агент без теста; осиротевший тест; агент с
локальной копией разбора `.sol`; агент/тест в обход `run_all`; висячая ссылка
`run_all`; воркфлоу не содержит `run_all --with-tests` или `test_run_all.py`;
две команды в одном шаге `run:`; команда лишь в комментарии; триггер-пути без
`ai-agents/**`; шаг `run:` без `- name:`, два шага с одинаковым `- name:` и
именованный шаг без тела `run:`/`uses:` —
мягко предупреждают, не валят) доказывает,
что страж реально краснеет (а на мягких проверках — лишь предупреждает), а на
чистом каталоге — зелёный (89/89). Включён в `run_all --with-tests`, поэтому при
добавлении агента без теста, копии разбора `.sol`, агента/теста в обход общего
прогона CI, выпадении `ai-agents/**` из триггеров воркфлоу или пропаже любой
обязательной команды воркфлоу — покраснеет.

Мета-агент `run_all` теперь **прогоняет стража заодно** и выводит его в сводке
отдельной графой: hard-«красное» стража валит общий вердикт, а его мягкие
предупреждения показывает строкой `мягких предупреждений стража: N` и под ней
построчным списком `⚠ [проверка] шаг: проблема` — так предупреждения видны и
объяснены в операторской сводке, не валя сборку.

## Рельсы (для всех агентов этого каталога)

- Никаких приватных ключей/секретов/реальных средств; никакого mainnet.

- Приватность заявителей (без персональных данных).
- Двуязычность RU↔EN документации в одном коммите.
