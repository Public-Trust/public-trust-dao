## Что уже есть

| [`Timelock.sol`](contracts/Timelock.sol) | Обязательная задержка между «решение принято» и «казна исполнила» (окно на аудит/апелляцию). `schedule`/`execute` — только `governor` (= Governor), `cancel` — только `guardian` (аварийное вето). Назначается `executor`'ом у `Treasury`/`Disbursement`. | каркас + тесты ✅ |
| [`scripts/deploy.js`](scripts/deploy.js) | **Сборка всего контура** одним проходом: разворачивает 5 контрактов и связывает их (executor=Timelock, governor=Governor, `renounceAdmin`, Reputation.governor=Timelock). После проводки ни один человек не двигает средства единолично. | скрипт + интеграционный тест ✅ |

### Конституционные свойства, заложенные в `Treasury` (и проверенные тестами)

- **Никто не владеет казной единолично** (ст. 1–2): двигать средства может только
  `executor`, которым по конституции должен быть мультисиг 3-из-5 / Timelock, а не
  один человек. Деплойер после конструктора **не имеет привилегий**.
- **Безопасность ≠ власть:** `guardian` может только поставить аварийную паузу, но
  **не может двигать средства**; `executor` не может ставить паузу (роли разделены).
- **Лимит/поэтапность** ([`ANTI-ABUSE`](../docs/ANTI-ABUSE.md) §1): потолок одной
  выплаты `maxRelease`; крупнее — дробить на транши.
- **Прозрачность** (ст. 3): каждое движение (депозит/выплата/пауза/смена роли)
  эмитит публичное событие; у выплаты есть `registryRef` — ссылка на запись
  решения в публичном реестре [`governance/registry/`](../governance/registry/).
- **Путь децентрализации** ([`GOVERNANCE`](../docs/GOVERNANCE.md), фазы A→D):
  `setExecutor` позволяет передать исполнение от bootstrap-мультисига к
  голосуемому Timelock — только текущим `executor`.

### Конституционные свойства, заложенные в `Disbursement` (целевой escrow)

Реализует [`ESCROW-TARGETED-DISBURSEMENT.md`](../docs/ESCROW-TARGETED-DISBURSEMENT.md)
(интерфейс `ITargetedDisbursement`):

- **«Не даём деньги — оплачиваем нужду» (целевой расход, ст. 5):** при `open` адрес
  поставщика услуги фиксируется в кейсе; `release(id, amount)` **не принимает адрес
  получателя вовсе** — транш уходит строго этому поставщику. Получатель помощи
  физически не может перенаправить средства себе.
- **Возврат — только в казну:** если услуга не оказана, `refund(id)` возвращает
  остаток на адрес `treasury` (казна под коллективным контролем), **не** получателю.
- **Поэтапность** ([`ANTI-ABUSE`](../docs/ANTI-ABUSE.md) §1–§2): помесячные транши
  через накопление `released` + потолок одного транша `maxRelease`, а не аванс.
- **Обеспеченность escrow:** нельзя открыть кейс без свободного баланса
  (`escrowedTotal ≤ balance`) — учёт `available()`.
- **Те же роли, что у казны:** двигать средства может только `executor`
  (мультисиг/Timelock); `guardian` только ставит паузу; события на каждую ветвь
  (`open`/`release`/`refund`/`pause`) + `registryRef` для сверки с реестром.

### Конституционные свойства, заложенные в `Reputation` (бейдж участника)

Реализует [`GOVERNANCE.md`](../docs/GOVERNANCE.md) §2–§3 («один человек — один голос»,
«уникальность ≠ власть»):

- **«1 человек = 1 голос», не плутократия (ст. 2, запрет №5):** `votingUnits(addr)`
  возвращает `0` для не-участника и `1 + min(reputationPoints, reputationCap)` для
  участника. Вес всегда в коридоре `[1 .. 1+cap]` — **власть денег невозможна**.
- **Soulbound (непередаваем):** у контракта **нет ни одной функции перевода**
  (`transfer`/`approve`/`transferFrom`) — by design. Право голоса нельзя продать,
  подарить или сосредоточить скупкой (GOVERNANCE.md §2 «НЕ продаваемое право»).
- **Уникальность ≠ власть:** `verifier` лишь подтверждает уникальность человека
  (`mint`/`revoke` бейджа), `governor` правит параметры (репутация/потолок/роли).
  **Ни одна роль не двигает средства** — в контракте их нет; решает голосование.
- **Жёсткий потолок множителя:** бонус за вклад обрезается `reputationCap`, чтобы не
  возникла новая элита; потолок и репутацию меняет только `governor` (голосованием).
- **Прозрачность** (ст. 3): каждое изменение (выдача/отзыв/репутация/смена роли)
  эмитит событие с `registryRef`; передача ролей (`setVerifier`/`setGovernor`) —
  путь децентрализации (фазы A→D).

### Конституционные свойства, заложенные в `Governor` + `Timelock` (управление)

Реализует [`GOVERNANCE.md`](../docs/GOVERNANCE.md) §4–§7 (Governor → Timelock →
Treasury/Disbursement — прямое голосование, исполняемое с обязательной задержкой):

- **«1 человек = 1 голос», не плутократия (ст. 2, запрет №5):** вес голоса берётся
  из [`Reputation.votingUnits`](contracts/Reputation.sol); не-участник не голосует и
  не вносит предложений. Порог внесения — равный для всех (членство), **не денежный**.
- **Governor сам средства не двигает (ст. 4):** принятое решение ставится в очередь
  `Timelock`, а казна (`Treasury`/`Disbursement`) исполняет лишь то, что прислал
  `Timelock` после прошедшего голосования. `execute` раньше срока невозможно.
- **Задержка = окно на аудит/апелляцию/вето** ([`ANTI-ABUSE`](../docs/ANTI-ABUSE.md)):
  между «принято» и «исполнено» стоит обязательная пауза `minDelay`; в ней `guardian`
  может аварийно отменить операцию (`cancel`) — **вето, не распоряжение средствами**.
- **Разделение ролей:** `governor` (= контракт Governor) планирует/исполняет;
  `guardian` только аварийно отменяет; `admin` — разовый bootstrap-конфигуратор и
  **обязан отказаться от прав** (`renounceAdmin`); ни одна роль не двигает казну.
- **Параметры — только голосованием (§6, путь A→D):** срок/кворум/задержка и смена
  ролей после renounce меняются **только через сам механизм** (`onlyTimelock`/
  `onlySelf`), не в обход. Подсчёт кворума и большинства публичен и детерминирован.

### Сборка контура: `scripts/deploy.js` (часть 3c)

Скрипт разворачивает все пять контрактов и связывает их в единый механизм по
[`GOVERNANCE.md`](../docs/GOVERNANCE.md) §4–§7:

```
  Reputation ──(вес «1 человек = 1 голос»)──▶ Governor
                                                 │ queue/execute
                                                 ▼
  guardian ──(аварийное вето)──▶ Timelock ──(executor)──▶ Treasury
                                    ▲                     Disbursement
                                    └ governor = Governor
```

Итог проводки (проверяется интеграционным тестом):
`Treasury.executor` = `Disbursement.executor` = **Timelock**; `Timelock.governor`
= **Governor**; `Timelock.admin` = **0** (bootstrap снят через `renounceAdmin`);
`Reputation.governor` = **Timelock** (параметры голоса меняет только голосование).
После проводки деплойер не сохраняет привилегий — **никто не двигает средства
единолично** (ст. 1–2). Роли `guardian`/`verifier` на реальном деплое задаются
через окружение `GUARDIAN_ADDRESS`/`VERIFIER_ADDRESS`; на локали подставляются
тестовые подписанты.

## Запуск (локально, без сети и денег)

```bash
cd contracts
npm install          # один раз
npm test             # компиляция + все тесты на встроенной сети Hardhat
npm run deploy:local # развернуть и связать весь контур на in-process сети (демо)
```

CI прогоняет тесты на каждый push/PR ([`.github/workflows/contracts.yml`](../.github/workflows/contracts.yml)).
Интеграционный сценарий «заявка → голос → Timelock → целевая выплата поставщику»
покрыт в [`test/Integration.test.js`](test/Integration.test.js) (через `Disbursement`).

## Стек

[Hardhat](https://hardhat.org) + ethers v6 + chai. Solidity `0.8.24`. Тесты идут
на встроенной in-process сети Hardhat — **никаких внешних RPC, ключей, mainnet**.
Конфиг деплоя на testnet (напр. Polygon Amoy, chainId 80002) — закомментированный
шаблон в [`hardhat.config.js`](hardhat.config.js); ключи подставляет оператор
через `contracts/.env` (в `.gitignore`), не в репозиторий.
