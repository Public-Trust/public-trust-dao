# Публичный реестр решений — Public Trust DAO

Это **публичный, проверяемый, защищённый от подмены** журнал решений и операций
DAO. Он реализует конституционные принципы **прозрачности** и **неизменяемости
записей** ([`docs/CONSTITUTION.md`](../../docs/CONSTITUTION.md),
[`docs/PRINCIPLES.md`](../../docs/PRINCIPLES.md)) — ещё до IPFS-пиннинга
(следующий шаг Этапа 3).

> Это часть инфраструктуры общественного блага. Реестр **не** является
> финансовым инструментом и **не** обещает доходность. Записи о выплатах в
> примерах — тестовые (testnet), без реальных средств.

## Как это работает: hash-chain (цепочка хешей)

Каждая запись — отдельный JSON-файл в [`records/`](records/). Записи
**добавляются только в конец** (append-only) и **криптографически сцеплены**:

- у каждой записи есть `prev_hash` — sha256 предыдущей записи;
- у генезис-записи (`seq=0`) `prev_hash` = 64 нуля;
- `hash` записи = sha256 её **канонической формы без поля `hash`**.

Изменение любой старой записи меняет её `hash`, рвёт `prev_hash` следующей —
и `verify` это сразу обнаруживает. Историю нельзя переписать незаметно.

### Каноническая форма (воспроизводимо кем угодно)

```
canonical = json.dumps(record_без_поля_hash,
                        sort_keys=True,
                        separators=(",", ":"),
                        ensure_ascii=False)
hash = sha256(canonical.encode("utf-8")).hexdigest()
```

Любой может независимо пересчитать хеши и проверить целостность всей цепочки.

## Структура записи

Конверт описан схемой [`schema/record.schema.json`](schema/record.schema.json):

| поле | смысл |
|------|-------|
| `seq` | монотонный номер с 0 |
| `id` | стабильный идентификатор, напр. `PTD-0001` |
| `type` | `genesis`/`decision`/`disbursement`/`governance`/`audit`/`appeal`/`reputation`/`note` |
| `timestamp` | момент в UTC, ISO-8601 (`…Z`) |
| `actor` | кто инициировал (роль/агент/хранитель), **без персональных данных заявителей** |
| `summary` | краткое человекочитаемое описание |
| `refs` | ссылки: статьи конституции, Snapshot-пропозалы, tx-хеши, IPFS CID |
| `payload` | тип-специфичные данные (схемы по типам — в [`schema/`](schema/)) |
| `prev_hash` | sha256 предыдущей записи |
| `hash` | sha256 канонической формы этой записи |

Payload для `disbursement` (распределение средств) описан
[`schema/disbursement.schema.json`](schema/disbursement.schema.json) и
отражает требования [`docs/PRIORITIES.md`](../../docs/PRIORITIES.md) и
[`docs/ANTI-ABUSE.md`](../../docs/ANTI-ABUSE.md): приоритет нужды, поэтапные
выплаты, лимиты, коллективная проверка, окно апелляции.

## Приватность

В реестр **не** попадают персональные данные заявителей. Только псевдонимные
`case_id` и проверяемые ссылки. Прозрачность операций ≠ раскрытие людей,
которым помогают.

## Инструмент: `scripts/registry.py`

Без сторонних зависимостей (только стандартная библиотека Python) — поэтому
проверка запускается одинаково локально и в CI.

```bash
# проверить схему, цепочку хешей и индекс (exit!=0 при любой проблеме)
python3 scripts/registry.py verify

# добавить запись (seq, prev_hash, hash считаются автоматически)
python3 scripts/registry.py append \
  --type decision \
  --id PTD-0003 --slug short-slug \
  --summary "Краткое описание" \
  --actor "роль/агент" \
  --timestamp "2026-06-22T12:00:00Z" \
  --ref "docs/CONSTITUTION.md" \
  --payload /path/to/payload.json

# пересобрать index.json из записей
python3 scripts/registry.py reindex
```

Машиночитаемый индекс — [`index.json`](index.json): `count`, `head_hash` и список
записей. `head_hash` — «отпечаток» всего реестра на данный момент; его удобно
пиннить в IPFS и публиковать.

## Гарантия в CI

Workflow [`.github/workflows/registry.yml`](../../.github/workflows/registry.yml)
прогоняет `verify` на каждый push/PR. Любая подмена истории или рассинхрон
индекса валит сборку — целостность реестра проверяется автоматически и публично.
