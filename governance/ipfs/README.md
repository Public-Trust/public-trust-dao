# IPFS — постоянные content-addressed ссылки на ключевые документы

Этот каталог даёт **проверяемые постоянные идентификаторы (IPFS CID)** для
нормативных документов Public Trust DAO. Цель — прозрачность: у каждого
ключевого документа есть идентификатор, который вычисляется **из самого
содержимого** и проверяется кем угодно **без доверия к нам** и **без аккаунта**.

## Что здесь

- `manifest.json` — детерминированный список ключевых документов с их размером,
  `sha256` и IPFS `CID` (CIDv1). Машиночитаемый, отсортированный, воспроизводимый.
- Генератор/проверка: [`scripts/ipfs_manifest.py`](../../scripts/ipfs_manifest.py).

В манифест входят все нормативные доки (`docs/*.md`) и машиночитаемый индекс
[реестра решений](../registry/) (`governance/registry/index.json`). Индекс
реестра содержит `head_hash` — отпечаток всей hash-chain журнала решений, поэтому
один CID на `index.json` косвенно фиксирует и весь реестр.

## Как устроен CID (и почему ему можно доверять)

Для файла, помещающегося в один блок IPFS (≤ 256 KiB при дефолтном чанкере),
`ipfs add --cid-version=1 --raw-leaves=true` возвращает **CIDv1** с кодеком
`raw` (0x55), мультихеш которого — это просто **sha2-256** от байт файла:

```
CIDv1 = multibase_base32( 0x01 0x55 0x12 0x20 <sha256(file_bytes)> )
        (version=1)(codec=raw)(mh=sha2-256)(len=32)
```

Отсюда два независимых способа проверки:

1. **Через IPFS** (если установлен `ipfs`/Kubo):
   ```
   ipfs add --cid-version=1 --raw-leaves=true --quiet docs/CONSTITUTION.md
   ```
   Вывод обязан совпасть с `cid` для этого файла в `manifest.json`.

2. **Без IPFS, чистым пересчётом** — `scripts/ipfs_manifest.py` делает ровно это
   стандартной библиотекой Python (без зависимостей):
   ```
   python3 scripts/ipfs_manifest.py verify
   ```
   Команда пересчитывает размер/sha256/CID каждого файла и сверяет с манифестом.
   Любое расхождение (подмена документа или ручная правка манифеста) → ненулевой
   код возврата. Эта же проверка крутится в CI на каждый push/PR.

Файлы крупнее одного блока в манифесте честно помечаются
`single_block: false` / `cid: null` (`cid_kind: needs-unixfs-chunking`) — мы
никогда не публикуем CID, который не можем вычислить точно. Сейчас все ключевые
документы одноблочные.

## Пиннинг (persistence) — нужен оператор

Вычисление и проверка CID **не требуют** ничего, кроме содержимого файлов.
Аккаунт пиннера нужен **только** для persistence — чтобы блоки реально жили в
сети IPFS. Когда у оператора появится аккаунт (web3.storage / Pinata /
nft.storage) и токен в `.env` (НЕ в репозитории), команды пиннинга:

```
python3 scripts/ipfs_manifest.py pin-commands
```

CID, полученные после пиннинга, **обязаны совпасть** со значениями в
`manifest.json` — иначе пиннился не тот контент. См. раздел «НУЖНО ОТ ОПЕРАТОРА»
в [`PROGRESS.md`](../../PROGRESS.md).

## Приватность

В манифест входят только **публичные нормативные документы** и индекс реестра.
Никаких персональных данных заявителей — реестр оперирует псевдонимными
`case_id` (см. [`governance/registry/README.md`](../registry/README.md)).
