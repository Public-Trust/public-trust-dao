# DECISIONS — Public Trust DAO

Значимые решения и их причины. Кратко, по делу.

## 2026-06-22 — Архитектура автономного контура (bootstrap)

- **Пульс = systemd + loop.sh, отдельно от агента-строителя.** Сердцебиение
  не должно зависеть от решений самого строителя. Агенту запрещено править
  `loop.sh`/`report.sh`/сервис (мандат в BUILDER.md) — иначе одна плохая правка
  убивает непрерывность. Паттерн повторяет «пульс» autonomous-trader.
- **Отчёт по git, а не по самоотчёту агента.** loop.sh считает реальные коммиты
  (`BEFORE..AFTER`) и шлёт их в Telegram. «Готово» = измеренный коммит, не слова
  агента. Снижает риск ложных «сделал».
- **Один шаг за сессию.** timeout 1800s на сессию + один связный кусок плана →
  частые проверяемые коммиты вместо больших нестабильных заходов.
- **Токены только в окружении/.env, git-auth через credential.helper из env.**
  Токен не пишется в .git/config (helper читает $GH_TOKEN при push). .env в
  .gitignore, chmod 600.
- **bypassPermissions + skip-permissions для headless-петли.** Иначе claude -p
  виснет на запросах разрешений. Поведение ограничено мандатом BUILDER.md и
  жёсткими рельсами безопасности (testnet-first, без реальных денег).
- **Рельса денег — единственная защёлка, не тормоз стройки.** Контракты пишутся
  и тестируются полностью; реальные средства — только после аудита + одобрения
  оператора + Safe multisig 3-из-5 людей.

## 2026-06-22 — Этап 1, Фундамент: документы и лицензии

- **Отдельные нормативные доки, а не только MANIFESTO.** Манифест — цельный
  north star, но контрактам и агентам нужны узкие, ссылаемые по статьям
  спецификации. Поэтому `CONSTITUTION.md` (статьи), `PRIORITIES.md` (+ JSON для
  кода), `ANTI-ABUSE.md` (требования к контрактам/агентам), `PRINCIPLES.md`
  (запреты буквально). Дублирование намеренное и функциональное: один источник
  смысла (манифест) → нормативные срезы для реализации.
- **Лицензия кода = AGPL-3.0 (не MIT).** Мандат допускал MIT или AGPL. Выбран
  AGPL: сетевой копилефт не даёт закрыть проект даже при SaaS-развёртывании —
  это прямо реализует конституционный принцип «всё открыто, ничего не скрывать»
  и защиту от закрытых форков. Цена — меньшая совместимость с проприетарным
  переиспользованием, что для общественного блага приемлемо.
- **Лицензия документов = CC-BY-SA-4.0.** Свободное распространение конституции/
  манифеста при сохранении авторства и копилефта.
- **loop.sh с правкой IS_SANDBOX не трогаю (повтор, сессия 2).** Незакоммиченная
  правка пульса по-прежнему висит в `git status` — это сердцебиение, мандат
  запрещает. В свои коммиты не включаю (стейджу пути точечно, не `git add -A`).

## 2026-06-22 — Этап 2, Публичный сайт

- **Самодостаточный статический сайт, без внешних запросов.** Ни Google Fonts,
  ни CDN, ни аналитики: системные шрифты, локальный CSS. Для проекта про
  прозрачность сайт, который тайком пингует сторонние сервисы и собирает данные
  посетителей, — противоречие. Бонус: работает офлайн и при пиннинге в IPFS.
- **Деплой через GitHub Actions, источник `/web`.** Pages умеет «root» или
  «/docs», но в `/docs` лежат нормативные .md — мешать их с сайтом не хочу.
  Actions-артефакт из `/web` разделяет «исходники доков» и «собранный сайт» и
  даёт `.nojekyll` (отдаём готовый HTML без обработки Jekyll).
- **Нормативные доки линкуются на github.com, не дублируются в сайт.** Один
  источник истины (.md в репо, рендерится GitHub'ом), сайт — витрина. Меньше
  риск рассинхрона текста конституции между сайтом и репо.
- **Включение Pages — действие оператора.** Source=«GitHub Actions» ставится в
  UI репозитория один раз; агент это сделать не может. Записано в PROGRESS →
  «НУЖНО ОТ ОПЕРАТОРА», стройку не блокирует (артефакт всё равно собирается). Незакоммиченная правка пульса
  (нужна для bypassPermissions под root) — часть сердцебиения; мандат запрещает
  править loop.sh. Оставляю как есть, в свои коммиты не включаю.

## 2026-06-22 — Этап 3, Прозрачность: реестр решений

- **Hash-chain поверх git, а не вместо него.** Git и так хранит историю, но
  переписываемую (force-push, rebase). Реестр — это append-only журнал, где
  каждая запись сцеплена sha256 с предыдущей: подмена старой записи рвёт цепочку
  и ловится `verify`. Это даёт «неизменяемость записей» (конституция) на уровне
  содержимого, независимо от git, и готовит почву под IPFS (пиннуем `head_hash`).
- **Инструмент без сторонних зависимостей.** `registry.py` использует только
  стандартную библиотеку (свой мини-валидатор подмножества JSON Schema). Причина:
  `verify` должен запускаться одинаково и детерминированно в любом окружении и в
  CI без установки пакетов — целостность не должна зависеть от доступности pip.
- **Догфудинг записей через сам тул.** Genesis и примеры созданы командой
  `append`, а не вручную, — так хеши гарантированно корректны и формат проверен
  «изнутри». Подмена в середине цепочки протестирована → verify exit=1.
- **Приватность важнее полноты.** В реестр не пишем персональные данные
  заявителей — только псевдонимный `case_id` и проверяемые ссылки. Прозрачность
  операций ≠ раскрытие уязвимых людей (дух конституции о достоинстве).
- **CI как публичная гарантия.** `registry.yml` гоняет verify на каждый push/PR:
  подмена истории или рассинхрон index.json валят сборку публично.
- **JSON Schema-файлы остаются как контракт.** Тул валидирует по
  `record.schema.json`; схема `disbursement.schema.json` пока документная (payload
  свободной формы в конверте), но привязана к PRIORITIES/ANTI-ABUSE и будет
  применяться по типам, когда подключим выплаты на testnet (Этап 5).

## 2026-06-22 — Этап 3, прозрачность ч.2: IPFS-манифест (вычисляемые CID)

- **CID вычисляем локально, без аккаунта пиннера.** Для одноблочных файлов
  (<=256 KiB) `ipfs add --cid-version=1 --raw-leaves=true` даёт CIDv1 raw, чей
  мультихеш — это sha2-256 содержимого. Значит `scripts/ipfs_manifest.py` считает
  настоящий, проверяемый CID чистой стандартной библиотекой — без демона IPFS и
  без токена. Аккаунт пиннера нужен ТОЛЬКО для persistence, не для адресации.
  Это сильнее «sha256-манифеста» из плана: публикуем сразу боевые IPFS-ссылки.
- **Корректность подтверждена эталоном.** Реализация воспроизводит канонический
  CID пустого файла `bafkreihdwdcefgh4dqkjv67uzcmw7ojee6xedzdetojuzjevtenxquvyku`
  — общеизвестную константу, поэтому формуле CID можно доверять.
- **Никогда не публикуем CID, который не можем вычислить точно.** Файлы крупнее
  блока честно помечаются `single_block:false` / `cid:null` (нужен UnixFS-chunking).
  Сейчас все ключевые доки одноблочные.
- **manifest_fingerprint** — sha256 канонического списка (path+sha256+cid): один
  отпечаток всего набора, удобно якорить/сравнивать. `verify` ловит и подмену
  файла, и ручную правку манифеста (протестировано → exit=1).
- **CI `ipfs.yml`** гоняет verify на каждый push/PR, затрагивающий docs/реестр/
  манифест: контент и опубликованные CID не могут разойтись незаметно.
